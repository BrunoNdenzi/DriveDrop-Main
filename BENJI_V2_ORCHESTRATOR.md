## 1. Benji Orchestrator — Full Responsibility Map

### 1.1 What the Orchestrator Is

`BenjiOrchestrator` is a **single class** that is the only permitted entry point for all AI-driven operations. Every channel (chat, SMS, API, voice, webhook) produces a canonical `OrchestratorRequest` before any AI reasoning occurs.

```
backend/src/services/benji/BenjiOrchestrator.ts     ← NEW
```

No service in the system is allowed to call OpenAI, select a tool, or mutate shipment state without going through `BenjiOrchestrator.process()`.

**What the orchestrator does NOT do:**
- It does not re-implement pricing math (delegate to `tool:pricing.calculate`)
- It does not re-implement route TSP (delegate to `tool:route.optimize`)
- It does not re-implement heuristic scoring (delegate to `tool:dispatch.*`)
- It does not own I/O with Twilio or Supabase (delegate to tools)

### 1.2 Canonical Request/Response Contracts

```typescript
// ─── Canonical Input ──────────────────────────────────────────────────────

type InputChannel = 'chat' | 'sms' | 'api' | 'voice' | 'webhook';
type UserType     = 'client' | 'driver' | 'admin' | 'broker';

interface OrchestratorRequest {
  requestId:       string;           // UUID — generated by the channel adapter
  inputChannel:    InputChannel;
  userId?:         string;           // resolved from JWT or phone → profile lookup
  sessionId?:      string;           // stable across messages in same conversation
  userType?:       UserType;
  rawInput:        string;           // original text, exactly as received
  structuredInput?: Record<string, unknown>; // pre-parsed (API body, webhook payload)
  context?: {
    currentPage?:  string;
    shipmentId?:   string;
    attachments?:  string[];         // Supabase storage URLs
    phoneNumber?:  string;           // present for sms channel
    voiceCallId?:  string;           // present for voice channel
  };
  metadata?:       Record<string, unknown>;
}

// ─── Canonical Output ─────────────────────────────────────────────────────

type OrchestratorActionType =
  | 'SHIPMENT_CREATED'        | 'SHIPMENT_UPDATED'       | 'SHIPMENT_STATUS_REPLY'
  | 'QUOTE_RETURNED'          | 'LOAD_ACCEPTED'           | 'LOAD_RECOMMENDATIONS_RETURNED'
  | 'ROUTE_OPTIMIZED'         | 'DISPATCH_ANALYZED'       | 'DISPATCH_ASSIGNED'
  | 'DOCUMENT_EXTRACTED'      | 'CHAT_REPLY'              | 'SMS_REPLY'
  | 'CLARIFICATION_REQUESTED' | 'ESCALATED_TO_HUMAN'      | 'NO_OP';

interface OrchestratorAction {
  type:    OrchestratorActionType;
  payload: Record<string, unknown>;  // action-specific data (shipmentId, price, etc.)
}

interface OrchestratorResponse {
  requestId:               string;
  intent:                  BenjiIntent;
  intentConfidence:        number;             // 0–1
  action:                  OrchestratorAction;
  reply:                   string;             // human-readable, channel-formatted
  replyFormatted?: {
    chat?:  { message: string; suggestions: string[] };
    sms?:   { body: string };                  // ≤ 160 chars per segment
    voice?: { ssml: string };
  };
  requiresClarification:   boolean;
  clarificationQuestions?: string[];
  toolsUsed:               string[];           // e.g. ['tool:memory.read', 'tool:shipment.create']
  toolResults?:            Record<string, ToolResult<unknown>>;
  validationResult:        ValidationResult;
  memoryContext:           MemoryContext;      // what memory was injected
  eventsEmitted:           string[];           // event IDs written to benji_events
  latencyMs:               number;
  usage?:                  TokenUsage;
}
```

### 1.3 Orchestrator Responsibilities (Ordered)

```
RECEIVE       Normalize input from any channel into OrchestratorRequest
AUTHORIZE     Verify userId, userType, channel permissions
BUDGET CHECK  Reject if user's daily token budget exhausted (from benji_preferences)
MEMORY QUERY  Load user's benji_memory entries (non-blocking, 500ms timeout)
INTENT        Classify intent (keyword → GPT-4o-mini fallback)
CONTEXT BUILD Assemble full context object for tool selection
TOOL SELECT   Decide which tool(s) are needed (deterministic decision tree, no LLM)
TOOL EXEC     Execute selected tools sequentially (tools cannot call each other)
VALIDATE      Pass all tool outputs through DecisionValidator (HARD GATE)
DECIDE        If approved → format action output; if rejected → clarification flow
EMIT EVENTS   Write all decision events to benji_events (fire-and-forget)
MEMORY WRITE  Persist any newly inferred facts to benji_memory (fire-and-forget)
RESPOND       Return OrchestratorResponse to channel adapter
```

### 1.4 Orchestrator Tool Selection Logic

Tool selection is a **deterministic decision tree** — no LLM involved at this step. The intent classification result drives it:

```typescript
// backend/src/services/benji/BenjiOrchestrator.ts (selection logic)

function selectTools(intent: BenjiIntent, context: OrchestratorRequest): string[] {
  // Memory read is ALWAYS first (if userId is known)
  const base = context.userId ? ['tool:memory.read'] : [];

  const toolMap: Record<BenjiIntent, string[]> = {
    create_shipment:    [...base, 'tool:shipment.parse', 'tool:pricing.calculate', 'tool:shipment.create'],
    track_shipment:     [...base, 'tool:shipment.lookup'],
    get_quote:          [...base, 'tool:pricing.calculate'],
    find_load:          [...base, 'tool:dispatch.recommendations'],
    accept_load:        [...base, 'tool:dispatch.accept'],
    route_optimize:     [...base, 'tool:dispatch.recommendations', 'tool:route.optimize'],
    dispatch_analyze:   [...base, 'tool:dispatch.analyze'],
    dispatch_assign:    [...base, 'tool:dispatch.assign'],
    upload_document:    [...base, 'tool:document.extract'],
    account_help:       [...base, 'tool:chat.respond'],
    payment_help:       [...base, 'tool:shipment.lookup', 'tool:chat.respond'],
    faq:                [...base, 'tool:chat.respond'],
    escalate_human:     [],       // no tools — immediate escalation
    other:              [...base, 'tool:chat.respond'],
  };

  return toolMap[intent] ?? [...base, 'tool:chat.respond'];
}
```

**Constraints enforced at this level:**
- Drivers can never receive `tool:dispatch.analyze` or `tool:dispatch.assign` (admin-only)
- Clients can never receive `tool:dispatch.*` tools
- `tool:dispatch.accept` requires `userType === 'driver'` and the requesting userId must match the driverId in context

### 2.3 Tool Execution Rules (Enforced by Registry)

```typescript
class BenjiToolRegistry {
  private tools: Map<string, BenjiTool<unknown, unknown>> = new Map();

  async execute<T>(
    toolName: string,
    input: unknown,
    context: ToolContext
  ): Promise<ToolResult<T>> {
    const tool = this.tools.get(toolName);
    if (!tool) return { success: false, error: `Unknown tool: ${toolName}`, latencyMs: 0 };

    // Rule: validate role permissions
    if (tool.requiredRole && !tool.requiredRole.includes(context.userType as UserRole)) {
      return { success: false, error: `Role ${context.userType} cannot execute ${toolName}`, latencyMs: 0 };
    }

    const start = Date.now();
    try {
      const result = await Promise.race([
        tool.execute(input as never, context),
        new Promise<never>((_, reject) =>
          setTimeout(() => reject(new Error('tool_timeout')), tool.timeoutMs ?? 15_000)
        ),
      ]);
      return { ...result, latencyMs: Date.now() - start };
    } catch (err) {
      // Rule: tools never throw — registry catches and wraps
      return { success: false, error: String(err), latencyMs: Date.now() - start };
    }
  }
}
```

## 3. Decision Validation Layer

### 3.1 Purpose

The validator is a **hard gate** between tool execution and action commitment. It runs after tools produce their output, before any state-mutating action is committed. It cannot be bypassed.

```
backend/src/services/benji/DecisionValidator.ts   ← NEW
```

---

### 3.2 Validation Result Contract

```typescript
interface ValidationResult {
  approved:       boolean;
  reason?:        string;        // human-readable reason for rejection
  missingFields?: string[];      // fields that must be provided before retry
  riskLevel:      'low' | 'medium' | 'high';
  blockedBy?:     string;        // which rule ID caused rejection
  canRetry:       boolean;       // false = permanent block (e.g. unauthorized role)
}
```

---

### 3.3 Validation Rules by Action Type

#### `SHIPMENT_CREATE`
| Rule ID | Condition | Block Reason |
|---|---|---|
| `SC-001` | `parsedData.pickup_location` is null or empty | Missing pickup location |
| `SC-002` | `parsedData.delivery_location` is null or empty | Missing delivery location |
| `SC-003` | `parsedData.vehicle.year` is null or NaN | Missing vehicle year |
| `SC-004` | `parsedData.vehicle.make` is null or empty | Missing vehicle make or model |
| `SC-005` | `confidence < 0.60` | Confidence too low — clarification needed |
| `SC-006` | `userId` is null | Not authenticated |
| `SC-007` | `parsedData.pickup_location === parsedData.delivery_location` | Pickup and delivery are the same location |
| `SC-008` | `estimatedPrice < 0 OR estimatedPrice > 50000` | Price out of acceptable range — manual review |
| `SC-009` | `userType === 'driver'` | Drivers cannot create client shipments |

**Risk levels:**
- `SC-005`, `SC-008`: `medium`
- `SC-006`, `SC-009`: `high`
- All others: `low`

---

#### `LOAD_ACCEPT`
| Rule ID | Condition | Block Reason |
|---|---|---|
| `LA-001` | Shipment status ≠ `PENDING` | Shipment is no longer available |
| `LA-002` | `driverId ≠ requesting userId` | Cannot accept on behalf of another driver |
| `LA-003` | `driver.is_verified ≠ true` | Driver account not verified |
| `LA-004` | Driver already has an `ASSIGNED` shipment with overlapping pickup window | Scheduling conflict |
| `LA-005` | Shipment already has a `driver_id` set | Already assigned |

**Risk levels:** All `LA-*`: `high` (state mutation)

---

#### `DISPATCH_ASSIGN`
| Rule ID | Condition | Block Reason |
|---|---|---|
| `DA-001` | `userType ≠ 'admin'` | Requires admin role |
| `DA-002` | Any `matchScore < 60` in assignment batch | Below minimum confidence threshold |
| `DA-003` | Any shipment in batch has `status ≠ PENDING` | Stale analysis — rerun required |
| `DA-004` | Any driverId appears more than once in batch | Duplicate assignment — greedy error |
| `DA-005` | `assignments.length > 50` | Batch too large — split into ≤ 50 |

**Risk levels:** `DA-001`: `high`; `DA-003`, `DA-004`: `medium`; others: `low`

---

#### `ROUTE_OPTIMIZE`
| Rule ID | Condition | Block Reason |
|---|---|---|
| `RO-001` | `stops.length < 2` | Need at least 2 stops to optimize |
| `RO-002` | Any stop missing `lat`/`lng` | Invalid stop coordinates |
| `RO-003` | `stops.length > 20` | Too many stops for TSP algorithm (O(n²) cost) |
| `RO-004` | Any `timeWindow.latest < now()` | Expired time window in stop list |

**Risk levels:** All `RO-*`: `low` (read-only)

### 3.4 Validation Execution

```typescript
class DecisionValidator {
  async validate(
    actionType: OrchestratorActionType,
    toolOutputs: Record<string, ToolResult<unknown>>,
    context: OrchestratorRequest
  ): Promise<ValidationResult> {
    const rules = VALIDATION_RULES[actionType] ?? [];
    const failures: Array<{ ruleId: string; reason: string; riskLevel: string }> = [];

    for (const rule of rules) {
      const passed = await rule.check(toolOutputs, context);
      if (!passed) failures.push({ ruleId: rule.id, reason: rule.reason, riskLevel: rule.risk });
    }

    if (failures.length === 0) {
      return { approved: true, riskLevel: 'low', canRetry: true };
    }

    const highestRisk = failures.some(f => f.riskLevel === 'high') ? 'high'
                      : failures.some(f => f.riskLevel === 'medium') ? 'medium' : 'low';

    return {
      approved:      false,
      reason:        failures[0]!.reason,
      missingFields: failures.flatMap(f => f.missingFields ?? []),
      riskLevel:     highestRisk,
      blockedBy:     failures[0]!.ruleId,
      canRetry:      !failures.some(f => f.ruleId === 'SC-009' || f.ruleId === 'DA-001'),
    };
  }
}
```

**If validation fails → orchestrator routes to clarification flow:**

```typescript
// Clarification flow (inside BenjiOrchestrator)
if (!validationResult.approved) {
  const questions = buildClarificationQuestions(validationResult);
  await emitEvent('validation_failed', { actionType, blockedBy: validationResult.blockedBy });

  return {
    ...baseResponse,
    action: { type: 'CLARIFICATION_REQUESTED', payload: { validationResult } },
    reply: buildClarificationReply(validationResult, context.inputChannel),
    requiresClarification: true,
    clarificationQuestions: questions,
  };
  // NOTE: NO state mutation occurs. Validation failure = zero side effects.
}
```

## 6. SMS Orchestration Flow

### 6.1 Why SMS Is a First-Class Channel

SMS is how drivers communicate while driving. It must support the full orchestration pipeline with a 160-char constraint on output. No feature available in chat should be unavailable over SMS.

**Inbound webhook:** `POST /api/v1/benji/sms/webhook`  
**Authentication:** Twilio HMAC signature verification (`X-Twilio-Signature` header)  
**No JWT required** — user resolved by phone number lookup.

### 6.4 SMS Clarification Flow

When validation requires clarification, Benji sends a numbered question over SMS:

```
Benji:  "To create your shipment I need a few details:"
        "1. What city is the vehicle being picked up from?"

Driver: "Charlotte NC"

Benji:  "Got it. Charlotte NC pickup."
        "2. Where is it going?"

Driver: "Miami FL"

Benji:  "Perfect. What year/make/model is the vehicle?"
```

SMS conversation state is maintained in `benji_conversations` with `sessionId = 'sms:+1XXX'`. The orchestrator detects multi-turn SMS clarification via `requiresClarification = true` in the previous session's last message.

### 7.3 Event Replayability

Events are designed to be replayable. The `benji_events` table stores enough information to reconstruct what happened without re-executing the action.

**Replay rules:**
1. `intent_detected`, `tool_selected`, `memory_queried`, `prompt_rendered` → safe to replay (read-only)
2. `tool_executed` where tool is read-only → safe to replay
3. `validation_passed`, `validation_failed` → safe to replay (no side effects)
4. `shipment_created`, `load_accepted`, `dispatch_auto_assigned` → **NOT safe to replay** without idempotency key check
5. `sms_sent` → **NOT safe to replay** — guard with `MessageSid` deduplication

**Idempotency for state-mutating events:**
```sql
-- Add idempotency key to benji_events
ALTER TABLE benji_events ADD COLUMN idempotency_key text;
CREATE UNIQUE INDEX benji_events_idempotency_idx ON benji_events (idempotency_key)
  WHERE idempotency_key IS NOT NULL;
```

`idempotency_key` = `requestId + ':' + actionType` for all mutation events.

### 7.4 Event Audit Query Examples

```sql
-- Full decision trace for a single request
SELECT event_type, source, payload, created_at
FROM benji_events
WHERE payload->>'requestId' = 'req-uuid-here'
ORDER BY created_at;

-- All validation failures in the last 24 hours
SELECT payload->>'blockedBy' as rule, COUNT(*) as failures
FROM benji_events
WHERE event_type = 'validation_failed'
  AND created_at > now() - interval '24 hours'
GROUP BY 1 ORDER BY 2 DESC;

-- Tools that timeout most often
SELECT payload->>'toolName' as tool,
       COUNT(*) FILTER (WHERE NOT (payload->>'success')::boolean) as failures,
       AVG((payload->>'latencyMs')::int) as avg_latency_ms
FROM benji_events
WHERE event_type = 'tool_executed'
  AND created_at > now() - interval '7 days'
GROUP BY 1 ORDER BY 2 DESC;

-- SMS delivery rate
SELECT
  COUNT(*) FILTER (WHERE event_type = 'sms_sent') as sent,
  COUNT(*) FILTER (WHERE event_type = 'sms_received') as received,
  COUNT(*) FILTER (WHERE event_type = 'sms_sent'
    AND payload->>'status' = 'failed') as failed
FROM benji_events
WHERE created_at > now() - interval '7 days';

-- Memory hit rate (how often memory is found vs empty)
SELECT
  AVG((payload->>'memoriesFound')::int) as avg_memories_found,
  COUNT(*) FILTER (WHERE (payload->>'timedOut')::boolean = true) as timeouts,
  COUNT(*) FILTER (WHERE (payload->>'memoriesFound')::int = 0) as empty_memory
FROM benji_events
WHERE event_type = 'memory_queried'
  AND created_at > now() - interval '7 days';
```
